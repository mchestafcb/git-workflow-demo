The right way to think of Git

  • If you can think of something you want it to do, it does it
  • Don't be afraid

Dev/Stage/Master

  • Dev = Try things out
  • Stage = Stable, share with the team/client
  • Master = Production

Feature branching

  • As is common practice, we use the feature branching model.
  • Great for teams
  • Always work on a feature branch, then merge into dev
  • Name the branches after the feature, not the author
  • Keep small feature branches local, push larger ones to github
  • Hot fixes branch off of master, then merge into master and dev

Commits

  • Cheap. Use often.

Commit Messages

  • Summarize clearly the problem being solved.
  • Does not have to be one line.
  • Jokes are fun, but not helpful.

Merging

  • Resolve manual conflicts. Look for: <<<<<<
  • Use git diff, view diffs on Github or use Diffmerge
  • When in doubt, file a pull request. A way to discuss/review before merging
